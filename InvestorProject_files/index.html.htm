4.Key Design Decisions

Design Decisions

Description

Investor Login uses custom secure implementation of Login and password and does not use Azure AD B2C for secure login and authentication

The Investor login supports login using email, PAN number and mobile no.

Using PAN for login mandates the storage of PAN number in Azure AD B2C as a custom attribute etc. Since these needs careful planning and potential exposure of PAN number to Azure AD Operators and users, it is decided NOT to use Azure AD B2C for login for now.

Azure AD B2C will be considered in future if in case the Citizen services are exposed via SWS.

PAN number and all other PII data is encrypted and masked when stored to the Dataverse

Using Azure Blob Storage for storing Files/Documents of Investors

Though Dataverse provides options to store files, it is decided to use Azure Blob Storage for secure storage of files and documents.

Azure blob storage provides versioning and easy access to files while still ensuring that the documents are secure at storage and at transit.

Also provides optimized cost for Dataverse as Dataverse size does not increase due to file storage

Power Pages custom forms for UI development

Power Pages custom forms is used for UI development instead of Power Apps Canvas Apps and Model Driven Apps.

SWS Application is a customer facing application and not an internal application, with rich UI/UX with increased number of Forms and Fields to be added. Canvas Apps and Model driven apps are useful only for Internal non-customer facing Applications.

Authentication and Authorization between SWS and different departments are achieved using JWT Token Based Authentication and Basic Authentication

Azure Entra ID or other Commercial Single Sign On (SSO) are not used for Authentication and authorization between SWS and different Departments.

Some departments who use Older versions of DotNet which does not support JWT Token based authentication, follows basic authentication.

 

 

 

 

 

 

 

 

5.Solution Architecture

The below section describes the solution architecture, comprising of Logical architecture, data architecture, infrastructure/deployment architecture and security architecture.

5.1Logical Architecture

The Single Window Application is built into 3 logical layers Presentation Layer, Service Layer and a Data access Layer. The system also has an API Layer which exposes the system APIs to outside world and consumes the API calls from external systems. The logical Architecture is depicted below.

Figure 1 Logical Architecture

The solution is designed based on the principle of layered architecture, where solution components are grouped into logical layers.

The logical architecture includes the following layers:

Presentation Layer:

This layer comprises components responsible for implementing and displaying the user interface (UI). It includes controls for user input and display, as well as components that organize user interaction. User Interface components represent the visual elements of the application used to present information and receive user input. Power Pages will be used as technology in the presentation layer.

Service Layer:

The service layer in the architecture diagram provides a range of services crucial for the functionality and usability of the system. Here's a detailed explanation of each component within the service layer:

        Registration: This service handles the registration of new users or entities within the system, ensuring they can gain access and interact with the system's features.

        Combined Application Form: This service manages the initial approval processes, likely for applications or requests submitted by users. It is a preliminary step before full clearance is granted.

        Affidavit Based Clearance: This service processes affidavits submitted by users, verifying and clearing them based on specific criteria or requirements.

        Know Your Incentive/ Incentive Calculator: This tool allows users to understand the incentives available to them and calculate potential benefits based on their inputs or qualifications.

        Know Your Approval: This service provides users with the status of their approvals, ensuring they can track the progress of their applications or requests.

        Grievance Management: This service handles user grievances, providing a system for submitting, tracking, and resolving complaints or issues raised by users.

        MIS Reports: Management Information System (MIS) reports generate analytical and operational reports, providing insights and data for decision-making and performance tracking.

        Incentive: This service deals with the distribution and management of incentives, ensuring eligible users receive their benefits accordingly.

        GEN AI Chatbot: An artificial intelligence-powered chatbot designed to assist users with their queries, provide information, and facilitate interactions with the system.

6.      Core Business Service Layer:

The Core Business Service Layer handles fundamental system operations such as document management, workflow management, user management, and role and access management. These components ensure that the system functions smoothly and securely, supporting the services provided in the Service Layer.

  1. Document Management System:
    • Power Pages integrates with Dataverse and Azure Blob for document storage and management.
    • This can create pages that allow users to upload, view, and manage documents in a secure and compliant manner.
    • Custom workflows in Power Automate can be used for document approval and version control.
  2. Workflow Management:
    • Power Pages connects seamlessly with Power Automate to design and implement workflows.
    • This can automate business processes such as form submissions, approvals, and notifications by building custom workflows that interact with your backend systems.
  3. Role and Access Management:
    • Power Pages uses custom username/password based on Mobile No, PAN or email for investors to login to the system. These registered users are mapped to appropriate permissions within Power Pages and Dataverse ensuring secure access to data.
    • Azure AD is used for Authentication and Authorization of Backend system users such as MD, JD, DD.
    • Different user groups (e.g., admins, contributors, and viewers) can have different access levels to content and functionalities based on their roles.
    • Permissions can be set that control who can view, edit, or submit forms, access workflows, and see notifications.
  4. Alert / Notifications:
    • Through Power Automate, alerts and notifications can be triggered based on user actions or events in Power Pages.
    • Notifications can be sent via email, SMS, or app notifications to inform users of important updates, such as task completion, approvals, or issues requiring attention.
  5. Reporting Engine:
    • Power Pages can leverage Power BI for interactive dashboards and reports that visualize key business data.
    • These reports can be embedded within Power Pages portals, enabling users to monitor business metrics and outcomes in real time.
  6. Master Data Management:
    • Power Pages relies on Dataverse for managing master data, ensuring consistency and accuracy across applications.
    • This can manage master data entities like customers, vendors, or products, and these entities can be exposed to Power Pages for user interaction or data collection.

7.      Data Access Layer:

This layer will include components that abstract the logic required to access the underlying data stores. It centralizes common data access functionality, simplifying application configuration and maintenance. The business layer will communicate with the data stores through this layer.

The data access layer uses the full capabilities of Dataverse for implementing the necessary data store and retrieval services required by the Single Window System.

 

8.      Integration Layer

The Invester portal will utilize API gateway functionality through various channels. An Integration Layer ensures smooth integration between internal and external systems, while an API Manager handles the transformation, routing, and monitoring. Asynchronous communication enhances performance and responsiveness.

Secure connection between App Service and other Azure Components

The diagram illustrates a secure communication process between an App Service, App Registration, and Key Vault (could be any Azure components) using Azure. Below are the key components and their interactions:

Components:

  1. App Service: Represents the application that contains the code and a certificate for authentication.
  2. App Registration: Configured in Azure Active Directory for identity and authentication. It validates the App Service.
  3. Key Vault: A secure storage service in Azure used to manage secrets, encryption keys, and certificates.

Communication Flow:

  1. Authentication with Certificate:
    The App Service uses a certificate to authenticate itself with the App Registration.
  2. Token Generation:
    Once authenticated, the App Registration provides an access token to the App Service.
  3. Access Secret with Token:
    Using the token, the App Service securely communicates with the Key Vault to retrieve secrets (e.g., API keys or passwords).

 

 

 

 

 

o   Entra ID Authorization Process Diagram (Identity and Access Management)

This diagram illustrates the workflow and communication for identity management using Microsoft Entra ID with Privileged Identity Management (PIM). Entra ID implementation in SWS ensures the following

  1. User Interaction:
    • The user initiates an identity verification process.
    • Multi-Factor Authentication (MFA) is employed to ensure secure access.
  2. Privileged Identity Management (PIM):
    • Based on the users identity verification, their assigned Admin Profile (e.g., Global Admin, Read-only, Service Admin) is determined.
    • Alerts are triggered if there are changes or activities requiring attention.
  3. Configurations and Management:
    • Administrators configure and manage PIM settings, such as roles and permissions.
    • Activities such as monitoring, auditing, and generating access reports are conducted to ensure compliance and security.
  4. Communication Flow:
    • Users authenticate and receive access based on their roles.

Alerts and reports are communicated to administrators to maintain oversight

12.1 Database backup and restore approach

The Backup and Restore process in Power Pages primarily relies on the underlying Dataverse platform where data and configurations are stored.

Step

Description

How to Achieve

 

 

Enable Dataverse Backups

Power Pages stores data in Dataverse; enabling Dataverse backups ensures recovery.

Go to the Power Platform Admin Center.

Navigate to Environments.

Enable System Backups under settings.

System Backups

Automatic backups are created every 12 hours by Dataverse and retained for 730 days.

Dataverse automatically manages these backups.

Access via Power Platform Admin Center.

Manual Backups

Allows creating on demand backups for immediate needs.

Go to Power Platform Admin Center → Environments → Select Environment → Backup Now.

Restore a Backup

Restores the environment data to a specific backup point in time.

Go to Power Platform Admin Center → Select Environment → Restore.

Choose a backup version.

Backup Website Content

Power Pages site metadata, code, and design configurations need additional backup.

Export Power Pages website using Power Platform CLI or Solution Export.

Version Control

Manage changes to configurations and content.

Use Power Platform Solutions to package and export Power Pages changes.

Integrate with GIT for version control.

Test Restoration

Periodically test the backup and restore process to ensure reliability.

Restore the backup to a sandbox environment and verify all data and site functionality.
